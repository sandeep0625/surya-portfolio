AMD

Firmware QA Engineering Intern

Austin, TX • Server CPU validation

I worked in server-platform validation, where firmware builds, BIOS settings, Linux state, PCIe devices, and high-speed I/O tests all had to match before a result meant anything.

Click to expand technical details

Platform context

The work sat around server CPU validation platforms with PCIe switching paths, ConnectX adapters, RDMA traffic, NVMe/CXL context, BIOS configurations, and Linux-based test execution. My job was to help prove whether the platform behaved correctly across controlled firmware and BIOS build cadences.

Validation flow

I worked through Daily, Staging, Weekly, and PI-style validation runs. Before trusting results, I had to confirm the test setup, BIOS build, device visibility, link state, OS logs, command output, and expected traffic behavior.

Debug mindset

A failing test was not automatically a bug. I learned to separate setup mistakes, device/link issues, firmware changes, Linux-side problems, and real regressions by collecting evidence instead of guessing from one failure.

What I improved

I became faster at platform triage: checking configurations, reading logs, reproducing issues, comparing expected versus actual behavior, and writing useful JIRA notes that another engineer could actually use.

ITW

Electrical Engineering Intern

Troy, OH • Custom embedded sensing prototype

I worked on an embedded sensing prototype that combined a custom STM32H7 board, audio/sensor capture, firmware bring-up, lab debug, and DSP-style analysis.

Click to expand technical details

Project goal

The prototype explored whether audio/vibration behavior could be used to understand slicer blade sharpness and machine condition. The challenge was not just collecting a signal; it was building a repeatable hardware + firmware + analysis path around it.

Board bring-up

I worked with a custom STM32H753-based board, checked power rails, validated interfaces, and used lab tools to debug behavior across I2C, I2S, UART, ADC/audio paths, and assembly-related issues.

Firmware + data

The firmware handled peripheral setup, capture flow, communication, and analysis support. The signal-processing side involved FFT workflows, thresholds, smoothing, amplitude trends, and turning noisy measurements into something usable.

Big lesson

This internship made board bring-up feel real. A schematic can be logical, but the actual board still has to be powered, measured, probed, and debugged layer by layer before the system can be trusted.
